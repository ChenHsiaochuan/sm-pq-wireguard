  # v4 设计文档：SM4 替换 ChaCha20-Poly1305

  **版本**：v3 (SM2 + SM3) → v4 (SM2 + SM3 + SM4)
  **日期**：2026-05
  **状态**：设计阶段（代码未动）
  **目标**：把握手与传输层所有 ChaCha20-Poly1305 / ChaCha20 / XChaCha20-Poly1305
   替换为 SM4-GCM / SM4-CTR，实现全国密合规。

  ---

  ## 一、SM4 Crate 选型

  ### 1.1 候选调研

  | crate | 版本 | digest 依赖 | SM4-GCM 支持 | 备注 |
  |-------|------|------------|-------------|------|
  | `sm4` | 0.5 | digest 0.10 | 仅裸块 | 需自行封装 GCM |
  | `sm4-gcm` | 0.1.2 | 无 digest 依赖 | ✓ 原生 | API 最简洁 |

  ### 1.2 选定方案

  使用 `sm4-gcm = "0.1.2"`。

  **核心 API：**
  ```rust
  use sm4_gcm::{sm4_gcm_aad_encrypt, sm4_gcm_aad_decrypt, Sm4Key};

  // 加密
  let (ciphertext, tag) = sm4_gcm_aad_encrypt(
      &Sm4Key([u8; 16]),  // 128-bit 密钥
      &[u8; 12],          // 96-bit nonce
      aad: &[u8],         // 附加认证数据
      plaintext: &[u8],
  );

  // 解密
  let plaintext = sm4_gcm_aad_decrypt(
      &Sm4Key([u8; 16]),
      &[u8; 12],
      aad: &[u8],
      ciphertext: &[u8],
      tag: &[u8; 16],
  ) -> Option<Vec<u8>>;

  Cargo.toml 新增：
  sm4-gcm = "0.1.2"

  三、Nonce 管理设计（核心）

  3.1 SEAL/OPEN — 握手 AEAD（crypto_params.rs）

  现状：
  const ZERO_NONCE: [u8; 12] = [0u8; 12];
  // ChaCha20Poly1305::new(key).encrypt(nonce=ZERO_NONCE, aad, plaintext)

  设计决策： 直接替换为 SM4-GCM，nonce 保持全零12字节。

  安全性论证： 握手 AEAD 每次使用新的临时密钥（KDF2
  派生），每把密钥只用一次，ZERO_NONCE 不构成 nonce-reuse 风险。

  替换方案： SM4-GCM drop-in，key 截取 KDF 输出前 16 字节。

  ---
  3.2 CHACHA20! — 裸流加密（crypto_params.rs）

  现状： 用 ChaCha20 流加密包裹 KEM 密文（f_static_ct_pq 字段），无 MAC。

  设计决策： 替换为 SM4-CTR 模式。

  // v3
  ChaCha20::new(key_32b.into(), &ZERO_NONCE.into())
      .apply_keystream(&mut buf);

  // v4
  use sm4::cipher::{KeyIvInit, StreamCipher};
  use sm4::Sm4;
  type Sm4Ctr = ctr::Ctr128BE<Sm4>;
  let mut cipher = Sm4Ctr::new(&key_16b.into(), &ZERO_NONCE.into());
  cipher.apply_keystream(&mut buf);

  Key 来源： KDF1(0³², DH1) 输出32字节，取前16字节作为 SM4-CTR key。

  安全性论证： 同 SEAL/OPEN，每次 DH1 唯一，每把 CTR 密钥只用一次。

  ---
  3.3 XSEAL/XOPEN — Cookie Reply AEAD（macs.rs）

  现状：
  let f_nonce: XNonce = XNonce::from(random_nonce_24b);  // 24字节随机
  // XChaCha20Poly1305 加密 cookie

  设计决策： 使用 SM4-GCM，nonce 取随机24字节的前12字节。

  // v4
  let rand_bytes: [u8; 24] = random();         // 仍生成24字节（存于 f_nonce 
  字段）
  let nonce: [u8; 12] = rand_bytes[..12].try_into().unwrap();
  sm4_gcm_aad_encrypt(&Sm4Key(cookie_key_16b), &nonce, aad, pt)

  f_nonce 字段保留24字节： 协议消息格式不变（24字节 nonce 字段），SM4-GCM
  只用前12字节，后12字节填零或随机（不影响安全性）。

  安全性论证：
  - Cookie key 每120秒轮换（COOKIE_UPDATE_INTERVAL）
  - DoS 防护机制限制了 cookie 请求速率
  - 在120秒窗口内，12字节随机 nonce 碰撞概率 ≈ 2⁻⁴⁸，可忽略

  ---
  3.4 Router 传输层 AEAD（router/send.rs, router/receive.rs）

  现状（使用 ring 库）：
  use ring::aead::{Aad, LessSafeKey, Nonce, UnboundKey, CHACHA20_POLY1305};

  // Nonce 构造：[0,0,0,0] + f_counter(u64 LE) = 12字节
  let mut nonce_bytes = [0u8; 12];
  nonce_bytes[4..].copy_from_slice(header.f_counter.as_bytes());

  let key = UnboundKey::new(&CHACHA20_POLY1305, &keypair.send.key[..]).unwrap();
  let key = LessSafeKey::new(key);
  key.seal_in_place_append_tag(nonce, aad, buf)?;

  问题：ring 库无 SM4 实现。

  设计决策： 移除 ring 依赖，改用 sm4-gcm crate 直接调用。

  替换方案（send.rs）：
  use sm4_gcm::{sm4_gcm_aad_encrypt, Sm4Key};

  let mut nonce = [0u8; 12];
  nonce[4..].copy_from_slice(header.f_counter.as_bytes());  // 格式不变

  let sm4_key = Sm4Key(keypair.send.key);  // key 已是 [u8; 16]
  let (ciphertext, tag) = sm4_gcm_aad_encrypt(&sm4_key, &nonce, aad, plaintext);

  替换方案（receive.rs）：
  use sm4_gcm::{sm4_gcm_aad_decrypt, Sm4Key};

  let sm4_key = Sm4Key(keypair.recv.key);
  match sm4_gcm_aad_decrypt(&sm4_key, &nonce, aad, ciphertext, &tag) {
      Some(plaintext) => { /* 成功 */ }
      None => { /* 认证失败，丢包 */ }
  }

  Nonce 格式不变： 仍是 [0,0,0,0] + counter_8b，SM4-GCM 的12字节 nonce 与
  ChaCha20Poly1305 完全相同。

  ---
  四、密钥长度调整

  4.1 核心问题

  ChaCha20 使用256-bit密钥（[u8; 32]），SM4 使用128-bit密钥（[u8; 16]）。
  KDF2 输出两个 GenericArray<u8, U32>（各32字节），SM4只需取前16字节。

  4.2 调整策略

  KDF 输出截取（noise.rs）：
  // v3 (KDF2 给两把32字节密钥)
  let (key_recv, key_send) = KDF2(&ck, &[]);
  // key_recv 和 key_send 都是 GenericArray<u8, U32>

  // v4 (截取前16字节)
  let key_recv_bytes: [u8; 16] = key_recv[..16].try_into().unwrap();
  let key_send_bytes: [u8; 16] = key_send[..16].try_into().unwrap();

  KDF2 用于握手 SEAL/OPEN 时（key 派生）：
  // v3
  let (ck_new, k) = KDF2(&ck, dh_output);
  // k 是 [u8; 32]，直接传给 SEAL/OPEN

  // v4
  let (ck_new, k_full) = KDF2(&ck, dh_output);
  let k: [u8; 16] = k_full[..16].try_into().unwrap();
  // k 传给 SM4-GCM SEAL/OPEN

  4.3 需要修改的类型

  types.rs：
  // v3
  pub struct Key {
      pub key: [u8; 32],
      pub id: u32,
  }

  // v4
  pub struct Key {
      pub key: [u8; 16],
      pub id: u32,
  }

  macs.rs：
  // v3
  cookie_key: [u8; 32]

  // v4
  cookie_key: [u8; 16]

  八、风险点与注意事项

  8.1 ring 依赖的完整移除

  ring 可能被其他模块（wireguard/、wireguard_pq_star/）依赖。需确认：
  grep -r "ring" workspace/v4_sm4/artifacts_implementation/src/ --include="*.rs"
  若 wireguard_hybrid 之外还有 ring 依赖，只移除 wireguard_hybrid 内的调用，不从
   Cargo.toml 删除 ring。

  8.2 sm4-gcm Tag 处理

  ring 的 seal_in_place_append_tag 会把 tag 追加到 ciphertext buffer 末尾。
  sm4-gcm 返回 (ciphertext, tag) 分开。
  需确认 router 层 buffer 的写入方式，手动将 tag 追加。

  8.3 解密缓冲区布局

  ring 的 open_in_place 期望 ciphertext + tag 在同一 buffer。
  sm4-gcm 需要分别传入 ciphertext 和 tag（最后16字节）。
  需调整 receive.rs 的 buffer 切分逻辑：&buf[..len-16] 为密文，&buf[len-16..len]
   为 tag。

  8.4 测试用例中的硬编码常量

  tests.rs 中有 precomputed_chain_key 和 precomputed_hash 测试，
  这些常量在 v4 中会因协议字符串变化而失效，需重新计算并更新。

  ## 二、核心改动详解

  ### 2.1 `Cargo.toml`

  ```toml
  # 新增（v4）
  sm4-gcm = "0.1.2"   # SM4-GCM AEAD
  sm4 = "0.5"         # SM4 块加密器（用于 CTR 模式）
  ctr = "0.9"         # 通用 CTR 模式封装

  版本选择说明：
  - sm4-gcm 0.1.2 是目前 crates.io 上唯一能直接用的纯 SM4-GCM 实现
  - sm4 0.5 与 ctr 0.9 都使用 cipher 0.4 trait，互相兼容

  2.2 types.rs —— Key 长度变更

  // v3
  pub struct Key {
      pub key: [u8; 32],  // ChaCha20 256-bit key
      pub id: u32,
  }

  // v4
  pub struct Key {
      pub key: [u8; 16],  // SM4 128-bit key
      pub id: u32,
  }

  影响： 凡是构造 Key 的地方都需要把 32 字节的 KDF 输出截断到 16 字节。

  2.3 noise.rs —— 三个宏全部替换

  SEAL/OPEN（握手 AEAD）

  // v3
  ChaCha20Poly1305::new(GenericArray::from_slice($key))
      .encrypt(&ZERO_NONCE.into(), Payload { msg: $pt, aad: $ad })

  // v4
  let mut k16 = [0u8; 16];
  k16.copy_from_slice(&$key[..16]);
  let sm4_key = Sm4Key(k16);
  let ct_vec = sm4_gcm_aad_encrypt(&sm4_key, &ZERO_NONCE, $ad, $pt);
  $ct.copy_from_slice(&ct_vec);

  关键点： 12 字节 ZERO_NONCE 在 ChaCha20-Poly1305 与 SM4-GCM 中通用；密钥取 KDF
   输出前 16 字节。每次握手 KDF 派生新密钥，零 nonce 不构成重用风险。

  CHACHA20!（裸流加密，包裹 KEM 密文）

  // v3
  let mut cipher = ChaCha20::new($key.into(), &ZERO_NONCE.into());  // 32-byte 
  key, 12-byte nonce

  // v4
  type Sm4Ctr = ctr::Ctr128BE<Sm4>;
  const ZERO_IV_SM4: [u8; 16] = [0u8; 16];

  let mut k16 = [0u8; 16];
  k16.copy_from_slice(&$key[..16]);
  let mut cipher = Sm4Ctr::new((&k16).into(), (&ZERO_IV_SM4).into());
  cipher.apply_keystream(&mut buffer);

  关键差异： SM4 块大小 16 字节，CTR IV 也是 16 字节（vs ChaCha20 的 12 字节
  nonce）。

  Key 构造（KDF2 输出截断）

  // v3
  let (key_recv, key_send) = KDF2!(&ck, &[]);
  Key { key: key_send.into(), ... }   // GenericArray<u8, U32> -> [u8; 32]

  // v4
  let (key_recv, key_send) = KDF2!(&ck, &[]);
  let mut key_send_16 = [0u8; 16];
  let mut key_recv_16 = [0u8; 16];
  key_send_16.copy_from_slice(&key_send[..16]);
  key_recv_16.copy_from_slice(&key_recv[..16]);
  Key { key: key_send_16, ... }

  2.4 crypto_params.rs —— XSEAL/XOPEN（Cookie AEAD）

  // v3
  XChaCha20Poly1305::new(GenericArray::from_slice($key))   // 32-byte key
      .encrypt(GenericArray::from_slice($nonce), ...)      // 24-byte nonce

  // v4
  let mut key_bytes = [0u8; 16];
  key_bytes.copy_from_slice(&$key[..16]);
  let sm4_key = Sm4Key(key_bytes);
  let ct_vec = sm4_gcm_aad_encrypt(&sm4_key, &$nonce[..12], $ad, $pt);  // 取前 
  12 字节

  安全性论证： Cookie 报文 f_nonce 字段在线传输保留 24
  字节（协议格式不变），SM4-GCM 只用前 12 字节。Cookie key 每 120s 轮换 + DoS
  速率限制，12 字节随机 nonce 在窗口内碰撞概率 ≈ 2⁻⁴⁸，可忽略。

  2.5 router/send.rs / router/receive.rs —— 移除 ring AEAD

  ring 库无 SM4 支持，必须替换。但 nonce 构造（[0u8;4] + counter_8b = 12
  字节）与 SM4-GCM 完全兼容，仅替换加解密调用：

  // v3
  let key = LessSafeKey::new(
      UnboundKey::new(&CHACHA20_POLY1305, &keypair.send.key[..]).unwrap(),
  );
  let tag = key.seal_in_place_separate_tag(nonce, Aad::empty(), &mut
  packet[..tag_offset]).unwrap();
  packet[tag_offset..].copy_from_slice(tag.as_ref());

  // v4
  let sm4_key = Sm4Key(job.keypair.send.key);
  let ct_with_tag = sm4_gcm_aad_encrypt(&sm4_key, &nonce, &[],
  &packet[..tag_offset]);
  packet.copy_from_slice(&ct_with_tag);

  性能权衡： ring 是 in-place 加密；sm4-gcm 返回 Vec<u8>，多一次分配 +
  拷贝。可作为后续优化点。

  2.6 协议字符串变更

  v3: Noise_IKpsk2_SM2_ChaChaPoly_SM3
  v4: Noise_IKpsk2_SM2_SM4GCM_SM3

  ⚠️ 互通性影响： v4 与前三版均不可互通（协议字符串作为初始 chain key 输入）。

  一、SM3 替换 Blake2s 完成情况

  替换范围（在 WireGuard 协议中的位置）

  SM3 替换的是 WireGuard
  握手协议里的哈希原语层——这一层在协议中起骨架作用，覆盖三类用途：

  1. Chain Key 派生（HKDF 链）：握手中 4 次密钥混入操作，全部依赖 HMAC-Hash
  构造的 KDF
  2. Handshake Hash（H 值）：每一步握手把消息字段累积进运行哈希，用于绑定上下文
  3. MAC1 / MAC2（DoS 防护）：每条握手报文末尾的两个完整性认证标签

  简单说，所有"算摘要、派密钥、做认证"的操作底层都是 SM3 在跑。

  主要难点

  SM3 与 Blake2s 输出长度（256 bit）和块大小（64
  byte）虽然一致，但结构不兼容——Blake2s 是 BLAKE 家族（HAIFA
  wide-pipe），原生支持 keyed-hash 与变长输出；SM3 是 SHA-2 风格的
  Merkle-Damgård，没有这两个能力。

  最难的一点出在 MAC1/MAC2：原版直接用 Blake2s 的 keyed 模式 + 16
  字节变长输出，SM3 做不到。我们的解决方案是用 HMAC-SM3 截断到 16 
  字节作为等价替代——HMAC 是经过严格密码学分析的标准 keyed-hash 构造，截断后
  unforgeability 仍然成立，所以安全性等价但实现机制不同。

  这说明国密替换不是 1:1 的算法
  drop-in，而是涉及协议层面的等价性论证，具有研究价值。

  验收结果：16/16 单元测试通过，握手时延仅 +0.49%，报文大小完全不变。

  ---
  二、SM4 替换 ChaCha20-Poly1305（进行中）

  替换范围

  SM4 要替换的是 WireGuard 的对称加密层，覆盖四个不同的使用场景：

  1. 握手 AEAD：身份哈希、timestamp、empty 字段的认证加密
  2. KEM 密文包裹：用流密码包住后量子 KEM 的密文
  3. Cookie 回复 AEAD：DoS 防护机制中的密文封装
  4. 传输层数据加密：建链后所有数据包的逐包加密（router 层）

  主要难点

  难点 1：密钥长度从 256-bit 减半为 128-bit
  ChaCha20 用 32 字节密钥，SM4 只用 16 字节。意味着 KDF 输出的截取规则、所有
  Key.key
  字段类型、整个会话密钥结构都要重新设计，影响面波及类型层、握手层、路由层。

  难点 2：Nonce 域不匹配，且 nonce 不能像原版那样随机生成
  - XChaCha20-Poly1305 的 nonce 是 192-bit（24
  字节），可以直接随机生成不担心碰撞
  - SM4-GCM 的 nonce 只有 96-bit（12 字节），随机生成存在 nonce-reuse
  风险，一旦发生就直接破坏 GCM 的认证安全性

  四种使用场景的 nonce 来源各不相同（全零 / 计数器 / 随机），需要逐一论证安全性
  、设计独立的派生策略，避免引入比原版更弱的安全假设——这是 v4 真正的核心难点。

  难点 3：路由层依赖 ring 库，但 ring 不支持 SM4
  原版传输层用 ring 的 in-place AEAD 接口加密每一个数据包，性能极优。但 ring 是
  Google 维护的库，完全不包含国密算法。这意味着传输层热路径必须整体抽离
  ring，换成国密 crate，工程改造量大于前两阶段，且可能引入性能损失（in-place vs
  Vec 分配），需要后续单独评估甚至自实现优化。

  难点 4：协议字段格式取舍
  Cookie 报文的 nonce 字段在线上是 24 字节，SM4-GCM 只用 12
  字节——是改协议格式（破坏报文兼容）还是保留字段只用前 12 字节（浪费 12
  字节带宽但向后兼容），需要做工程权衡。

  当前进度：设计方案已完成（docs/DESIGN_SM4.md），nonce
  策略已逐场景论证，正在推进编码实现。预计 3-4
  周完成全部替换、单元测试与基准对比。
